# 스프라이트시트 생성 전면 개편 계획

> 상태: **계획 (미결정 항목 협의 후 착수)** · 작성 2026-05-28
> 범위: 사용자 보고 7개 이슈 — 셀 오버플로 / 방향 옵션 / 캐릭터-이펙트 분리 /
> 프레임 옵션 / 캐릭터 앵커 / 잔재 제거 일관성 / 이펙트 중앙.
> 백엔드(codex 프롬프트·normalize 후처리·MCP 스키마) ↔ 풀스택(API·React 패널)
> ↔ 시각·통합 검증 양쪽 걸침.

## 설계 원칙: 근본 해결

오류 항목(①⑤⑥⑦)의 해결은 **모델 출력에 의존하지 않는 결정적 후처리 + 구조화된
입력 계약**으로 한다. 이전 시도들이 프롬프트 강화 → 모델이 안 따라 재발한 패턴을
끊는다. 프롬프트 튜닝은 결정적 단계의 입력 분포를 좁히는 보조 역할로만 사용.

- **잘림/오버플로는 normalize 가 보장**: 모델이 셀을 넘게 그려도 **시트 전체에
  단일 축소율** 을 적용해 "모든 셀이 셀 안에 100% 포함" + "프레임 간 캐릭터
  크기 일관성" 을 결정적으로 성립.
- **이펙트/캐릭터 구분은 UI 의 명시 선택으로**: 자연어 키워드 추론은 폴백.
  코드가 subjectType 으로 분기 → 앵커·이펙트 가드·잔재 제거 등 모든 결정적 단계의
  입력 신호가 됨.
- **앵커/잔재 제거는 픽셀 계산**: footY 중앙값으로 캐릭터 ground line 강제,
  적응적 임계값으로 잔재 제거.
- **자연어 + UI 이중 진입**: 기존 채팅도 동작하되, 패널 사용 시엔 모호함 0.

## 이슈별 진단·해법

### ① 셀 영역을 벗어나는 문제 (재발)
**진단.** 이전 수정들은 (a) 프롬프트 컨테인먼트 강화, (b) 이펙트 세로 중앙
정렬, (c) normalize 컴포넌트 keep 영역 확장으로 일부 케이스를 잡았지만,
**모델이 프레임을 셀보다 크게 그리면 normalize 가 잘라내거나 인접 셀로
넘김** → 결국 사용자 눈에는 "잘렸다/오버플로"로 보임. 프롬프트 의존이라 확률적.

**권장 해법 (결정적 안전망 — 시트-전역 균일 축소).**
`normalizeSpritesheetCells` 에 **2-패스 축소** 도입.
1. **패스 1 (측정)**: 각 셀의 union bbox(컴포넌트 keep 후) 의 폭·높이를 모은다.
   시트 전체에서 `maxBbW = max(셀별 bbW)`, `maxBbH = max(셀별 bbH)`.
2. **패스 2 (균일 축소율 계산)**:
   `cellSafeW = cellW - 2*margin`, `cellSafeH = cellH - 2*margin`
   `scale = min(1, cellSafeW/maxBbW, cellSafeH/maxBbH)`
   → scale < 1 이면 모든 셀의 추출 콘텐츠를 **같은 비율 scale 로** sharp 로
   축소(aspect 보존, nearest 보간 — 픽셀아트 무뎌짐 최소). scale ≥ 1 이면
   축소 없음.
3. **패스 3 (정렬)**: 축소된 콘텐츠를 캐릭터=footY 중앙값(⑤) / 이펙트=셀 중앙(⑦)
   기준으로 배치.

**왜 셀-독립 축소가 아닌가 (사용자 지적, 2026-05-28).**
셀마다 다른 축소율을 쓰면 프레임마다 캐릭터 크기가 들쑥날쑥해져 애니메이션이
크게 깜빡인다("크기 일관성" 파괴). **단일 축소율을 시트 전체에 적용**해야
1) 캐릭터 모든 프레임·모든 방향에서 동일 크기, 2) 이펙트는 가장 큰 프레임이
셀에 딱 맞고 작은 프레임은 자연스럽게 작은 채로 유지(성장 애니메이션 의도
보존). 빈 셀은 그대로 두고 측정에서 제외.

**결정 사항.** 시트 전체 단일 scale. 보간은 nearest. scale < 0.5 같은 극단치는
프롬프트가 너무 어긋났다는 신호이므로 로그 경고(추후 자동 재생성 정책 고려 —
범위 외).

**의존.** ⑤(앵커) 와 같은 normalize 경로. 함께 변경.

### ② 캐릭터 방향 옵션 (1/2/4/8 방향)
**진단.** 현재 `make_spritesheet` 는 단일 액션을 N프레임으로만 표현. 게임
에셋에서 흔한 "방향별 스프라이트" (4방향 걷기 등) 지원 부재 — 사용자가
직접 R×C 를 지정해야 하고, 모델이 행을 방향으로 일관되게 안 채움.

**권장 해법 (UI + 백엔드 계약).**
- `make_spritesheet` 입력에 **`directions?: 1|2|4|8`** 파라미터 추가.
  - 1: 단일 방향(기존 동작)
  - 2: 좌·우 (또는 상·하 — ⑦ 결정 항목)
  - 4: 하·좌·우·상 (게임 관례)
  - 8: 8방향 (4방향 + 대각선)
- **레이아웃 컨벤션**: `rows = directions, cols = framesPerDirection`.
  예: 4방향 6프레임 걷기 → 4×6 그리드, 각 행이 한 방향의 6프레임.
- **프롬프트**: 행별 방향 라벨을 codex 프롬프트에 명시
  ("Row 1 (top): character facing **down** — 6 walk-cycle frames. Row 2:
  facing **left**. ...").
- **방향 일관성**: 각 행 모든 프레임은 같은 캐릭터, 같은 자세 사이클, 같은
  포즈 페이즈 정렬. 행 간엔 카메라 각도(방향)만 바뀜.

**결정 사항 (확정).**
- 2방향 = **좌·우** (게임 관례)
- 4방향 = 하·좌·우·상 (행 순서)
- 8방향 = down/down-left/left/up-left/up/up-right/right/down-right (시계 방향)

**의존.** ④(프레임 옵션 UI) 와 같은 패널. ③(캐릭터-이펙트 분리) 와 같은
"캐릭터" 카테고리.

### ③ 캐릭터 시트에 공격/스킬 이펙트가 끼어드는 문제
**진단.** 사용자가 "캐릭터 4프레임 걷기" / "캐릭터 공격" 등의 시트를 만들 때,
모델이 자체 판단으로 마법 오라·발광 무기·이펙트 입자·궤적을 추가. 캐릭터-only
시트가 더러워짐.

**핵심 원칙 (확정, 2026-05-28).**
**캐릭터 시트와 이펙트 시트는 항상 분리. 캐릭터 시트엔 어떤 액션(걷기·대기·
공격·스킬 캐스팅 모두 포함) 이라도 이펙트가 들어가지 않는다.** 2D 게임 파이프라인
관례 — 캐릭터(몸·동작)와 이펙트(VFX)는 별도 시트로 생성해 런타임에 합성/오버레이.

| 시트 종류 | subjectType | 내용 | 이펙트 |
|---|---|---|---|
| 캐릭터 시트 | `character` | 캐릭터 몸·동작만 (걷기·대기·공격 모션·스킬 시전 포즈) | ❌ **항상 금지** |
| 이펙트 시트 | `effect` | 이펙트/VFX 만 (슬래시 궤적·폭발·번개) | 캐릭터 ❌ |

→ 공격 동작 + 이펙트가 필요하면 **두 장 따로 생성**: ① 캐릭터 공격 모션 시트
(휘두르는 자세만), ② 슬래시/폭발 이펙트 시트 — 게임에서 합성. 재사용성 ↑
(같은 이펙트를 여러 캐릭터에 재활용).

**권장 해법.**
1. **subjectType 명시 입력**: 패널에서 `종류 = 캐릭터 | 이펙트` 양자택일.
   자연어 진입 시엔 키워드 추론 (캐릭터 명사 / 이펙트 명사). 모호하면
   character 폴백(보수적).
2. **액션 카테고리** (캐릭터 시트 안에서만, 모션 유형 묘사용 — 이펙트 유무와
   무관):
   - `locomotion`: walk / run / idle / jump / crouch / sneak / climb
   - `reaction`: hit / death / dodge / block / stagger
   - `combat`: attack / skill / spell / charge / projectile (이펙트 없는
     **캐릭터 동작만** — 칼 휘두름·주문 시전 포즈)
   - `gesture`: wave / nod / interact / victory / taunt
   - **`custom`**: 사용자 자유 입력(textarea). 프리셋에 없는 동작을 자연어로
     명세 (예: "방패로 막으면서 뒤로 한 걸음", "양손 도끼를 머리 위로 들어
     내리치기"). 이펙트 가드는 동일 강제 — 사용자가 텍스트에 "폭발"이라 써도
     캐릭터 시트면 폭발 이펙트는 그려지지 않고 모션만 그려짐.
   카테고리는 codex 프롬프트의 동작 묘사를 정밀하게 만들 뿐, **이펙트 가드는
   subjectType=character 이면 액션과 무관하게 항상 강제 주입.**
3. **프롬프트 가드 (캐릭터 시트 무조건)**:
   ```
   Character body ONLY — NO magic effects, NO auras, NO glowing weapons,
   NO spell particles, NO motion lines, NO trails, NO sparkles, NO smoke,
   NO impact effects, NO extra decorative elements. Render only the
   character's body performing the action (joint motion, weapon pose if any).
   Visual effects belong on a SEPARATE effect sprite sheet.
   ```
   combat 카테고리의 attack/skill 도 동일 가드 — 모션만, 이펙트 금지.

**예시.**
- "마법사 4프레임" → character (action 모호) → locomotion idle 폴백 → 이펙트 없음
- "마법사 걷기 4프레임" → character + walk → 이펙트 없음
- "마법사 공격 4프레임" → character + attack → **휘두르는 자세만, 이펙트 없음**
- "마법사 마법 시전 4프레임" → character + spell cast → **시전 포즈만, 이펙트 없음**
- "슬래시 이펙트 4프레임" → effect → 이펙트 시트 (캐릭터 없음, ⑦ 적용)
- "번개 이펙트 4프레임" → effect → 이펙트 시트

**결정 사항.** 캐릭터 시트는 어떤 카테고리든 이펙트 가드 강제. 액션 카테고리는
캐릭터 동작 묘사용 메타데이터일 뿐, 이펙트 가드를 풀지 않는다. 이펙트가 필요하면
사용자가 별도 이펙트 시트를 명시적으로 생성.

**의존.** ②(방향) 과 같은 패널. ④(프레임 옵션) 의 액션 선택과 통합.

### ④ 프레임 옵션 — 상황별 다양화
**진단.** 현재 프레임 수는 자연어로만 입력 → 오케스트레이터가 R×C 매핑
(4→2×2, 16→4×4 등). 사용자가 게임 에셋 흐름에 맞는 선택 UX 가 없음.

**권장 해법 (구조화된 패널).**
새 **"스프라이트시트 생성" 전용 패널** (현재 채팅 + 자연어 입력 외에 추가):
```
┌─ 스프라이트시트 생성 ───────────────────┐
│ [● 캐릭터  ○ 이펙트]   ← 종류 (subject)  │
│                                          │
│ [캐릭터일 때 — 캐릭터 몸/동작만, 이펙트 ❌]│
│   액션: [공격 ▾]   (walk/idle/attack/    │
│                    skill cast/.../커스텀)│
│   [액션=커스텀일 때 보임]                 │
│     동작 설명: ┌──────────────────────┐  │
│                │ 방패로 막으면서      │  │
│                │ 뒤로 한 걸음 물러나기 │  │
│                └──────────────────────┘  │
│     ⓘ 캐릭터 동작만 묘사 (이펙트 적어도   │
│        그려지지 않음 — 별도 시트로)       │
│   방향: [4방향 ▾] (1/2/4/8)              │
│   프레임/방향: [6 ▾] (4/6/8/12)          │
│   앵커: [자동 ▾] (자동/발/엉덩이/중앙/머리)│
│   → 그리드: 4×6                          │
│   ⓘ 공격/스킬도 캐릭터 모션만 — 이펙트는  │
│     별도 [이펙트] 시트로 생성하세요       │
│                                          │
│ [이펙트일 때 — VFX 만, 캐릭터 ❌]         │
│   이펙트 종류: [슬래시 ▾]                │
│   프레임 수: [4 ▾] (4/6/8/12/16)         │
│   레이아웃: [2×2 ▾]                       │
│                                          │
│ 설명(선택): [텍스트]                     │
│ 참조 이미지(선택): [드롭/업로드]         │
│ 배경: [● 투명  ○ 흰 배경]                │
│ 루프: [● ON]                             │
│ [ 미리보기 ]  [ 생성 ▸ ]                  │
└──────────────────────────────────────────┘
```
- 종류 선택이 캐릭터/이펙트별 옵션을 토글.
- 결과는 자연어로 결합해 기존 `make_spritesheet` 호출 (백엔드 변경 최소).

**결정 사항.**
- 프레임/방향(캐릭터 시트는 "방향당 프레임 수" — 총 프레임 = 방향 × 프레임당)
- 행 = 방향, 열 = 프레임/방향 (②와 동일 컨벤션)
- **액션은 프리셋(walk/idle/attack/... 18종) + 커스텀(자유 텍스트) 둘 다 지원**.
  커스텀은 textarea로 캐릭터 동작을 자연어 명세 — 이펙트 가드는 동일하게 강제.

**진입·위치 (확정).** 우측 전용 패널(editor 패턴 — MaskCanvas/LayerCanvas/
ReskinPanel 과 동일 자리). 진입점: (a) 채팅 결과 카드의 `▦ 스프라이트` 단축어,
(b) Composer 상단 신규 [스프라이트시트 생성] 버튼. 생성 후 결과는 채팅 타임라인에
일반 결과 카드로 표시(기존 흐름과 일관).

**의존.** ②③⑦ 모두 이 패널로 입력.

### ⑤ 캐릭터 앵커 문제
**진단.** 현재 `CHARACTER ANCHOR` 프롬프트는 "hip/waist 셀 중심 + feet 일관
ground line" 인데 모델이 셀별로 발 라인 미세 드리프트(키 변동·바닥 라인 위·아래
변동) 자주 발생. 프롬프트 의존 한계.

**권장 해법 (결정적 후처리).**
캐릭터 시트일 때 normalize 의 footY 정렬을 **모든 셀 공통 ground line** 으로
강화:
1. 각 셀에서 본체(main component) 의 footY 검출 (현재 로직 재사용).
2. 셀들의 footY 분포에서 **중앙값을 목표 ground line** 으로 선정 (이상치 영향
   축소).
3. 각 셀의 본체 footY 를 목표에 맞춰 정렬(현재는 셀 하단-padding 기준 → 셀 간
   일관성 부족). 모든 셀이 동일한 글로벌 ground line 에 발 정렬.
4. 방향 시트(②) 면 같은 행(같은 방향) 안에서만 정렬 — 다른 방향(예: 캐릭터가
   기울어진 상하 방향) 은 본체 모양이 달라 별도 라인.

**결정 사항.** 정렬 기준은 footY 중앙값(robust). 평균은 이상치에 약함.

**앵커 보정 옵션 (사용자 요청, 2026-05-28).** 자동 정렬이 항상 완벽하진 않으므로
사용자 제어 두 층 추가:
1. **생성 시 — 앵커 전략 선택** (패널): `자동`(기본: 캐릭터=발 라인, 이펙트=중앙)
   / `발 기준(ground line)` / **`엉덩이 기준(hip)`** / `중앙(center)` /
   `머리 기준(top)`. params 에 저장. normalize 가 이 값으로 세로 정렬 기준을 결정.
   - **엉덩이(hip)**: 스켈레톤 애니메이션의 루트 본 위치 — 걷기/공격 시 발은
     움직여도 골반은 안정적이라 캐릭터 정렬에 가장 견고. 스켈레톤이 없으므로
     휴리스틱으로 추정: `hipY ≈ footY - (footY - headTopY) × 0.45` (발에서 위로
     본체 높이의 ~45% 지점). 각 셀의 추정 hipY 를 공통 라인(중앙값)에 정렬.
   - 정렬 기준값: 발=footY median, 엉덩이=hipY median, 중앙=bboxCenterY median,
     머리=headTopY median.
2. **생성 후 — 수동 미세보정** (SpriteCanvas): 시트별(방향 시트면 행별) **세로/
   가로 오프셋 슬라이더**. 자동 정렬 잔차를 비파괴적으로 nudge. "적용" 시 셀별
   콘텐츠를 오프셋만큼 재배치해 새 generation 으로 저장(또는 원본 덮어쓰기 선택).
   - 비파괴 미리보기: 슬라이더 조정 → SpriteCanvas 가 즉시 재렌더(저장 전).
   - GIF 미리보기와 연동 — 보정 결과를 바로 애니메이션으로 확인.

**결정 사항(앵커 옵션).** 생성 시 전략은 5종(자동/발/**엉덩이**/중앙/머리). 수동
보정은 SpriteCanvas 에서 오프셋 슬라이더 + 적용 시 새 generation 저장(원본 보존).

**의존.** ①(scale-to-fit) 후 정렬. normalize 의 같은 셀 루프 안에서 처리.
수동 보정은 Phase 3(SpriteCanvas) 작업.

### ⑥ 잔재 제거(chroma-key)의 일관성
**진단.** 사용자 보고에 따르면 chroma-key 후 잔재(녹색 fringe / 회색 halo)가
어떤 시트엔 깨끗하고 어떤 시트엔 남음. 35b8632 에서 임계값을 조정 가능하게
노출했지만 **기본값이 모든 케이스에 적합하지 않다**. 원인 후보:
- 모델이 #00ff00 대신 #20cc28 같은 어두운 녹색을 그림(채도/명도 변동)
- 안티에일리어싱이 매우 두꺼운 경우 fringe 가 더 깊이 박힘
- 캐릭터 자체에 녹색 요소(녹색 옷·녹색 머리)가 있어 잘못 키 아웃되거나, 반대로
  보존되며 fringe 가 남음

**권장 해법 (적응적 + 진단).**
1. **자동 임계값 보정**: 첫 패스에서 시트의 "녹색 분포" 를 히스토그램으로 분석
   (Hue 100~140 범위의 채도·명도 분포) → 모델이 그린 실제 녹색 톤에 맞춰
   chromaKey 임계값 자동 조정.
2. **2단계 디스필(despill)**: 키 아웃 후 남는 fringe 픽셀의 그린 채널만 추가
   탈채도 → halo 제거.
3. **녹색 캐릭터 보호**: 콘텐츠 영역(main component) 내부 픽셀은 키-아웃에서
   제외(연결 컴포넌트 기준 inside-mask). 외부 fringe 만 처리.
4. **진단 로그**: 셀별 알파 분포·녹색 잔여량을 로그로 남겨 (사용자 피드백 시
   원인 식별 용이).

**결정 사항 (확정).**
- 자동 임계값을 기본으로, 슬라이더는 보정용으로 유지.
- **녹색 캐릭터 자동 폴백 Phase 1 포함**: 시트 분석에서 콘텐츠 영역의 dominant
  hue 가 녹색(100~140°) 이면 chroma 키 색을 **마젠타(#ff00ff)** 로 전환해
  재생성 — 또는 첫 생성 전에 프롬프트가 어떤 키를 쓸지 결정. **구현 방식**:
  생성 시점엔 사용자 프롬프트에서 "녹색"·"green"·"슬라임"·"slime"·
  "잎"·"이끼" 등 키워드 감지 → 마젠타 키 자동 선택. 사후 분석 폴백은 비용
  과다(재생성)라 키워드 기반으로만.

**의존.** 독립적, 다른 이슈와 같은 normalize/post-process 단계.

### ⑦ 이펙트 시트의 중앙 정렬
**진단.** 이미 구현됨 (이전 커밋 17b4e27): 이펙트로 분류된 시트는
`centerVertically=true` 로 normalize 가 bbox 를 셀 세로 중앙에 배치. 가로도
mainCenterX 로 중앙. **검증으로 PASS 확인 완료** (bboxOffsetY ≈ 0).

**권장 해법.** 현재 동작을 유지. ④의 새 패널에서 "이펙트" 선택 시 명시적으로
이 경로를 타도록 ②③의 subjectType 시그널을 일원화(자연어 키워드 추론 →
패널 선택의 명시적 신호로 격상).

**결정 사항.** subjectType="effect" 이면 항상 중앙. 키워드 추론은 자연어 입력
(패널 미사용) 시의 폴백으로만 유지.

## 기존 기능에 미치는 영향 (회귀 방지 체크리스트)

이번 개편은 normalize·chroma·프롬프트·MCP 스키마·UI 진입점을 건드리므로, 다음
기능들이 같은 코드 경로/데이터를 공유한다. 각 항목의 영향과 호환성 처리 방침.

### 직접 영향 (같은 코드 경로 공유)

**1. 리스킨 모드 c (2-이미지) — `reskin_image` 시트 경로**
- 공유 경로: `normalizeSpritesheetCells` (server.ts:587 호출).
- 영향: ① scale-to-fit, ⑤ footY 중앙값 ground line, ⑥ chroma 변경 모두 reskin
  시트 출력에도 적용됨.
- 호환성 방침: **input 시트의 subjectType 을 부모 generation 의 params 에서
  상속**. 부모가 character → 캐릭터 정렬, effect → 중앙 정렬. 부모 정보 없으면
  현재 동작(발-앵커) 유지.
- 검증: 기존 reskin 검증 자산(마법사 reskin 시트)으로 회귀 0 확인.

**2. 캐릭터 오버레이 — reskin 모드 c 의 사용 사례**
- 위 1과 동일 코드 경로. **추가 고려**: 베이스 시트가 방향별 멀티-행(4×6 등)
  이면 오버레이 후에도 같은 행=방향 구조 유지 — normalize 가 셀 독립 처리라
  자동 유지됨.
- 검증: 4방향 베이스 + 캐릭터 참조 1장.

**3. classifyAnchor / subjectType 전파**
- 현재: 키워드 기반(character/effect). 새 패널은 명시적 `subjectType` 을 전달.
- 변경: `make_spritesheet` 입력 스키마에 `subjectType?: "character" | "effect"`
  + `actionCategory?: "locomotion"|"reaction"|"combat"|"gesture"` 추가.
- 자연어 진입(패널 미사용) 호환: 키워드 추론을 폴백으로 유지(현재 코드 그대로).
- DB 영속: generation.params 에 `subjectType` 저장 → 리스킨/SpriteCanvas 등이
  재참조.

**4. 스프라이트시트 input_image_ids 첨부 흐름**
- 캐릭터 참조 이미지 첨부(`inputGenerationId`) 지원. 새 패널도 동일 첨부 지원
  필요(예: "이 캐릭터를 4방향 걷기 시트로").
- 변경: 패널에 참조 선택 슬롯(ReskinPanel 모드 c 와 동일 그리드) 재사용.

### 간접 영향 (UX/표시 레이어)

**5. SpriteCanvas (스프라이트 분할 + GIF 미리보기)**
- 현재: `detectSpriteGrid` 가 GCD 역산으로 정사각 셀 가정(min 64, max 512).
- 새 방향 시트: 4×6 / 4×8 등 행 ≠ 열 → 셀이 정사각 아닐 수도 있음(make_spritesheet
  rows=1 케이스처럼 width 1.5배 가능).
- 호환성:
  - `detectSpriteGrid` 는 정사각 가정 → 비정사각 셀은 감지 실패 가능. **해결**:
    `generation.params` 에 `rows`, `cols` 를 영속(현재 `seamlessLoop` 만 저장)
    → SpriteCanvas 가 params 우선, GCD 폴백.
  - **방향 시트 GIF 미리보기 UX**: 한 시트에 여러 방향이 있을 때 사용자가 어떤
    방향을 재생할지 선택 가능해야 함. 행 선택 드롭다운("방향: 하/좌/우/상") 추가.

**6. 프롬프트 프리셋 (PromptLibrary)**
- 프리셋이 suffix 로 결합("with magical glow", "ethereal aura" 등)될 때 ③ 의
  "캐릭터 시트엔 이펙트 금지" 가드와 충돌 가능.
- 호환성 방침: **사용자 명시 입력(프리셋·자유 텍스트)이 자동 가드보다 우선**.
  단, 새 패널에서 종류=캐릭터 + 액션=locomotion 선택 + suffix 가 명백한 이펙트
  키워드면 경고 배너 표시("이펙트 묘사가 캐릭터 시트에 포함됩니다").

**7. 시스템 오케스트레이터 라우팅**
- 현재: 자연어 → make_spritesheet 매핑, "uniform cells, consistent subject across
  frames" 보일러플레이트 주입.
- 변경: 새 옵션(`directions`, `subjectType`, `actionCategory`) 인식 가이드 추가.
  보일러플레이트는 이미 중립화(`consistent subject`) 됨.

**8. SQL 스키마 / 마이그레이션**
- `generation.params` 에 새 필드 저장(`subjectType`, `actionCategory`,
  `directions`, `actionName`, `rows`, `cols`). params 는 JSON 컬럼이라 스키마
  변경 불필요. 단, **읽는 쪽(SpriteCanvas, ReskinPanel)** 에서 신규 필드 안전
  파싱(optional 처리). 마이그레이션 무용.

### 영향 없음 (격리됨)

- **배치 생성 / 히스토리 비교 / 레이어 분리 (LayerCanvas) / 인페인트 /
  리사이즈 / 업스케일 / 배경 제거 / 외부 업로드 / 정밀 색교체(reskin b-precise)**:
  스프라이트시트 후처리 경로 미경유. 회귀 없음.
- **GIF 다운로드 (gif.js)**: 셀 추출 결과를 받아 인코딩만 — 셀 구조 변경에 따라
  미리보기 UX 만 영향(위 5). 인코딩 자체는 무변경.

## 페이즈 분할 (작업·검증 단위)

### Phase 1 — 결정적 후처리 강화 (백엔드 중심) [근본 해결의 핵심]
- ① **시트 전체 단일 축소율** scale-to-fit (normalize 2-패스: max bbox 측정 → 단일 scale 산출 → 모든 셀에 동일 적용. aspect 보존, nearest 보간. 프레임 간 캐릭터 크기 일관성 보존)
- ⑤ 캐릭터 footY 글로벌 ground line (normalize, footY 중앙값) + 앵커 전략
  파라미터(자동/발/중앙/머리). 수동 보정 UI 는 Phase 3.
- ⑥ chroma-key 적응적 임계값 + despill + 본체 보호 + **녹색 캐릭터 키워드
  → 마젠타 키 자동 폴백**
- ⑦ 기존 centerVertically 유지
- **호환성**: `generation.params` 에 `subjectType`/`rows`/`cols` 영속 저장 →
  SpriteCanvas·리스킨이 재참조. reskin 시 부모 subjectType 상속.
- **검증**: 합성 시트(결정적, codex 무사용) + 실제 1~2장(캐릭터/이펙트 각 1)
  + 녹색 캐릭터(마젠타 키) 1장. 회귀: 기존 reskin 시트 1장.

### Phase 2 — 의미 분리 (오케스트레이터·프롬프트)
- ③ subjectType=character|effect 양자택일. 캐릭터 시트엔 액션 무관하게 이펙트
  가드 무조건 주입. 이펙트는 별도 effect 시트.
- 액션 카테고리는 캐릭터 동작 묘사용 메타데이터(이펙트 가드 풀지 않음).
- 자연어 키워드 → subjectType 매핑 (패널 없을 때 폴백, 모호하면 character).
- **검증**: "마법사 걷기" / "마법사 공격" / "마법사 마법 시전" 모두 → 이펙트
  없는 캐릭터 시트. "슬래시 이펙트" → 캐릭터 없는 이펙트 시트.

### Phase 3 — UI 패널 (풀스택)
- ④ 스프라이트시트 생성 패널 (우측 전용, editor 패턴): 종류·액션·방향·프레임
  - 액션 = 프리셋 드롭다운 + **커스텀(textarea)** — 커스텀 선택 시 동작 설명 입력
- ② 방향 옵션 UI (1/2/4/8) + 행별 방향 프롬프트 생성
- 패널 → `make_spritesheet` 결합 메시지 빌더 (참조 첨부 슬롯 포함)
- ⑤ **앵커 수동 보정 UI**: SpriteCanvas 에 세로/가로 오프셋 슬라이더(방향 시트면
  행별) + 비파괴 미리보기 + "적용" 시 새 generation 저장(원본 보존).
- **호환성**: SpriteCanvas 에 **방향 행 선택** 드롭다운 추가 + `params.rows`
  /`cols` 우선 사용(GCD 폴백 유지). 비정사각 셀 지원.
- **검증**: 패널 입력으로 4방향 6프레임 걷기 시트 생성 → 행=방향, SpriteCanvas
  에서 방향별 GIF 재생 + 앵커 오프셋 보정 동작.

### Phase 4 (선택) — 잔재 제거 고급
